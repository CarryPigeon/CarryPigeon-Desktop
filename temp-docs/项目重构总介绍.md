# CarryPigeon-Desktop 重构总介绍

## 1. 背景与问题

当前项目已经形成了可运行的功能闭环，但 TS 侧存在以下结构问题：

1. 页面与 store 过重：单文件承担了 UI、业务编排、副作用与跨模块协调，阅读成本高，回归风险大。
2. 功能边界混杂：如 `files` 同时包含传输能力与消息渲染语义，`network` 与 `servers` 在连接域重复承担编排职责。
3. 扩展成本高：消息渲染路径对插件扩展不够抽象，新增消息类型容易把条件分支继续堆在页面层。
4. 目录层级不统一：Feature 内部组织风格不一致，导致协作时难以快速判断模块职责。

## 2. 重构目标

本次重构目标是“结构最优先”，在不改变业务语义的前提下提升可读性与可维护性：

1. 低耦合高内聚：按业务能力边界拆分模块，避免横向直连。
2. 明确扩展点：在消息流中建立统一消息抽象，支持插件能力平滑接入。
3. 统一组织方式：Feature 下采用“基础层目录 + 一级平铺 subfeature”模式。
4. 统一入口：跨模块协作仅通过 `api.ts`，不允许深路径依赖实现细节。
5. 停用 ESLint：不再依赖 ESLint 规则与检查作为质量门禁，并同步重写相关规范文档。

## 3. 目标架构（目录约束）

### 3.1 Feature 内目录约束

每个 feature 下仅以下目录为“基础层保留字”：

- `mock/`
- `test/`
- `domain/`
- `di/`
- `data/`
- `presentation/`

除上述目录外的一级目录，均视为 subfeature（直接平铺在 feature 根目录，不使用中间 `subfeatures/`）。

### 3.2 跨模块依赖规则

1. `app/*` 仅允许依赖 `@/features/<feature>/api`。
2. feature 之间仅允许依赖对方 `api.ts`。
3. subfeature 之间禁止直接依赖彼此实现，协作通过 feature 聚合 API 或显式 ports 完成。
4. `shared/*` 只提供技术能力，不承载业务规则。

## 4. 核心领域重组策略

### 4.1 `auth + user` 合并为 `account`

原因：认证、会话、当前用户本质同属身份域，合并后可以减少跨 feature 循环协作。

建议结构：

- `features/account/auth-flow`
- `features/account/session`
- `features/account/profile`
- `features/account/current-user`

### 4.2 `network + servers` 合并为 `server-connection`

原因：server 选择、TLS 策略、连接状态与 server info 同属“连接到某台服务器”的业务域。

建议结构：

- `features/server-connection/rack`
- `features/server-connection/context`
- `features/server-connection/connectivity`
- `features/server-connection/server-info`

### 4.3 `platform` 上移到 `shared/platform`

原因：窗口与桥接能力是纯平台能力，跨 feature 复用，放 `shared` 更符合边界。

### 4.4 `files` 拆分：业务语义下沉到 chat，传输能力上提共享

1. 文件消息语义（如 `[file:share_key]`、文件气泡渲染）迁入 `chat/message-flow/message`。
2. 上传下载与 URL 构建沉淀为 `shared/file-transfer` 能力。

## 5. Chat 的目标结构（关键）

`chat` 按业务能力划分为三个一级 subfeature：

1. `room-session`：会话上下文、连接阶段、恢复与补拉触发。
2. `message-flow`：消息列表、分页、发送、回复、删除、已读。
3. `room-governance`：成员/申请/禁言/管理员治理能力。

其中 `message-flow/message` 负责统一消息抽象（核心扩展点）：

- `MessageEnvelope`：统一消息外壳。
- `MessageContentModel`：统一内容语义（core/plugin/unknown/file-ref）。
- `MessageRenderModel`：统一渲染模型。
- `MessageRendererResolver`：根据 domain 与 plugin registry 进行渲染决策。

## 6. 消息抽象与插件扩展策略

在 `message-flow/message` 内建立稳定接口，使页面层不再写大量分支判断：

1. 输入：后端 DTO + domain registry + 会话上下文。
2. 输出：可直接渲染的 `MessageRenderModel`。
3. 插件扩展：新增 domain 时仅需注册 renderer/composer 与 resolver 策略，不改页面壳组件。

## 7. 不兼容策略说明

本次重构采用“不向前兼容旧内部路径”的策略：

1. 不保留旧内部 API 适配层。
2. 迁移完成后直接删除旧模块与旧路径。
3. 以一次性路径清理换取后续长期清晰性。

## 8. 工程门禁策略（停用 ESLint）

根据本次重构约束，项目将停止使用 ESLint 规则与检查：

1. 不再执行 ESLint 命令作为本地或 CI 守门步骤。
2. `package.json` 中移除或停用 ESLint 相关 scripts（如 `lint`、`lint:fix` 的 ESLint 部分）。
3. 移除 ESLint 配置与规则依赖（`eslint.config.ts`、`eslint-rules/*` 与 `eslint*` 包）。
4. 质量门禁改为：
   - `vue-tsc --noEmit`
   - 现有日志规范脚本（`scripts/check-log-standards.sh`）
   - Rust 规范脚本（`scripts/check-rust-standards.sh`）
   - 关键链路人工回归（登录/连接/聊天/插件/文件）
5. 同步重写所有引用 ESLint 的规范文档，确保文档与执行策略一致。

## 9. 成功判定标准

1. 页面组件变薄：页面只做装配，不承载核心业务流程。
2. 业务单元可定位：每个 subfeature 只有单一变化轴。
3. 依赖路径清晰：跨模块导入仅走 `api.ts`。
4. 消息扩展可演进：插件消息接入无需修改页面主分支。
5. 主链路无回归：登录、连接、聊天、插件、文件上传下载行为保持一致。
6. 项目中不再依赖 ESLint 规则与检查，文档与实际门禁策略一致。
