# CarryPigeon-Desktop 重构计划

## 1. 计划概览

本计划按“先门禁、后结构、再收口清理”的顺序推进，共 9 个阶段：

1. 基线冻结与范围确认
2. 停用 ESLint + 文档重写
3. 目录与边界骨架落地
4. `account` 合并（`auth + user`）
5. `server-connection` 合并（`network + servers`）
6. `chat` 消息流重构（含 `message-flow/message` 抽象）
7. `files` 语义拆分（语义下沉 + 传输上移）
8. `shared` 平台与基础能力收口
9. 清理与验收

## 2. 阶段计划

## 阶段 0：基线冻结与范围确认

目标：确保重构期间可对比行为，不引入功能漂移。

任务：

1. 固定主链路回归清单（登录、连接、聊天、插件、文件上传下载）。
2. 冻结新增需求，仅允许结构重构 PR。
3. 记录关键入口文件与当前行为说明。

产出：

1. 回归 checklist 文档。
2. 重构范围边界（不改协议语义，不改产品行为）。

---

## 阶段 1：停用 ESLint + 文档重写

目标：完全停止 ESLint 规则与检查，并同步修订规范文档，避免“执行策略与文档不一致”。

任务：

1. 停用或移除 `package.json` 中 ESLint 相关脚本：
   - `lint` 中的 ESLint 步骤
   - `lint:fix`
   - 其他直接或间接调用 ESLint 的脚本
2. 移除 ESLint 配置与规则文件：
   - `eslint.config.ts`
   - `eslint-rules/*`
3. 移除 ESLint 相关依赖（`eslint`、`@eslint/*`、`typescript-eslint`、`eslint-plugin-vue` 等）。
4. 调整质量门禁为：
   - `vue-tsc --noEmit`
   - `scripts/check-log-standards.sh`
   - `scripts/check-rust-standards.sh`
   - 主链路人工回归
5. 重写现有规范文档中 ESLint 相关内容：
   - `docs/code-conventions.md`
   - `docs/日志与注释规范.md`
   - `docs/客户端开发指南.md`
   - `docs/前端调试与Mock.md`
   - `docs/新Feature接入检查清单.md`

产出：

1. 项目中不再存在 ESLint 执行路径。
2. 规范文档不再要求 ESLint，且新门禁说明完整。

---

## 阶段 2：目录与边界骨架落地

目标：先建立目标目录和 API 边界，再迁移代码。

任务：

1. 在 `features/*` 创建新的一级 subfeature 目录（平铺，不使用 `subfeatures/`）。
2. 为每个 feature 新建/整理聚合入口 `api.ts`。
3. 为每个 subfeature 建立 `api.ts`，声明最小公开面。
4. 建立迁移期约束文档，明确“禁止跨 subfeature 深路径依赖”。

产出：

1. 新目录骨架。
2. 统一导入边界规范。

---

## 阶段 3：`account` 合并（`auth + user`）

目标：整合身份域，减少会话/用户上下文分散管理。

目标结构：

- `features/account/auth-flow`
- `features/account/session`
- `features/account/profile`
- `features/account/current-user`

迁移重点：

1. 将 `auth` 登录流程、验证码流程迁入 `account/auth-flow`。
2. 将会话恢复、token 刷新、登出迁入 `account/session`。
3. 将当前用户状态与用户资料迁入 `account/current-user` 与 `account/profile`。
4. 替换全仓 `auth/api` 与 `user/api` 的引用为 `account/api`。

完成条件：

1. `features/auth` 与 `features/user` 不再作为独立业务入口。
2. 登录成功后的用户刷新链路由 `account` 内部闭环处理。

---

## 阶段 4：`server-connection` 合并（`network + servers`）

目标：统一服务器连接域，避免 socket/TLS/connect/serverInfo 分散。

目标结构：

- `features/server-connection/rack`
- `features/server-connection/context`
- `features/server-connection/connectivity`
- `features/server-connection/server-info`

迁移重点：

1. `currentServerSocket`、rack 管理、TLS 配置迁入 `rack/context`。
2. `connectNow/connectWithRetry/connectionState` 迁入 `connectivity`。
3. `serverInfoStore` 迁入 `server-info`。
4. `startupSession`、登录页、聊天页改为依赖 `server-connection/api`。

完成条件：

1. 外部不再分别依赖 `features/network/api` 与 `features/servers/api`。
2. 连接相关流程在单一 feature 内闭环。

---

## 阶段 5：`chat` 重构（重点）

目标：将 `chat` 从“页面+巨型store”重构为按业务能力划分的可组合模块。

目标结构：

- `features/chat/room-session`
- `features/chat/message-flow`
- `features/chat/room-governance`

### 5.1 `message-flow/message` 抽象落地

新增目录：

- `features/chat/message-flow/message/domain`
- `features/chat/message-flow/message/application`
- `features/chat/message-flow/message/presentation`

新增核心模型：

1. `MessageEnvelope`
2. `MessageContentModel`
3. `MessageRenderModel`
4. `MessageRendererResolver`

迁移重点：

1. 现有 `CoreTextBubble`、未知 domain 渲染分支统一进入 `MessageContentHost`。
2. 页面层不再直接判断 `core/plugin/unknown` 三分支。
3. 插件渲染接入改为走 resolver，减少页面耦合。

### 5.2 旧模块拆分迁移

目标文件（示例）：

1. `chatStore.ts` 拆入 `room-session/message-flow/room-governance`。
2. `liveChatStore.ts` 与 `liveChat/*` 按职责拆分到三个 subfeature。
3. `MainPage.vue` 变为装配页，仅绑定 subfeature API。

完成条件：

1. `MainPage.vue` 无核心业务状态机。
2. `message-flow/message` 成为消息语义唯一入口。

---

## 阶段 6：`files` 语义拆分（按你要求）

目标：把“文件消息语义”从 `files` 移出，进入 `chat/message-flow/message`。

任务：

1. 迁移 `FileMessageBubble` 到 `chat/message-flow/message/presentation/components`。
2. 迁移 `[file:share_key]` token 解析到 `chat/message-flow/message/domain`。
3. `files/api.ts` 删除消息渲染导出，仅保留上传下载能力导出。
4. `ChatCenter` 改为仅依赖 `chat/message-flow/message` 的消息组件。

同时进行：

1. 抽取通用文件传输能力到 `shared/file-transfer`。
2. `files` 保留业务层上传交互编排（如任务 UI）。

完成条件：

1. `features/files` 不再出现消息渲染语义组件。
2. 文件 token 与附件展示在 `chat` 内闭环。

---

## 阶段 7：`shared` 能力收口

目标：把跨 feature 技术能力收口到 shared，减少重复实现。

任务：

1. `features/platform` 整体迁入 `shared/platform`。
2. `shared/net` 统一 transport/protocol/tls 入口。
3. `shared/file-transfer` 提供 upload/download/buildUrl 抽象。

完成条件：

1. `shared` 仅包含技术能力，不包含业务规则。
2. feature 通过 shared 抽象调用，不依赖具体平台细节。

---

## 阶段 8：清理与验收

目标：删除旧路径与死代码，形成最终结构。

任务：

1. 删除已迁移旧目录与旧文件。
2. 全仓导入路径清理（仅保留 `feature/api` 与 `subfeature/api`）。
3. 运行类型检查与核心回归。
4. 更新文档（架构、feature README、迁移说明）。

完成条件：

1. 无旧路径残留引用。
2. 类型检查通过。
3. 主链路回归通过。

## 3. 关键迁移映射（摘要）

1. `features/auth/*` + `features/user/*` -> `features/account/*`
2. `features/network/*` + `features/servers/*` -> `features/server-connection/*`
3. `features/files/presentation/components/FileMessageBubble.vue`
   -> `features/chat/message-flow/message/presentation/components/FileRefMessageBubble.vue`
4. `features/chat/presentation/utils/coreTextToken.ts`
   -> `features/chat/message-flow/message/domain/coreTextFileSyntax.ts`
5. `features/platform/*` -> `shared/platform/*`

## 4. 回归测试建议（每阶段必做）

1. 登录与会话恢复：验证码登录、token 刷新、退出后状态清理。
2. 连接链路：切换服务器、TLS 策略、连接失败重试。
3. 聊天主链路：频道切换、消息发送、删除、回复、已读。
4. 插件链路：插件中心加载、启停、未知 domain 降级。
5. 文件链路：上传后 token 插入、消息渲染、下载打开。

## 5. 风险与控制

1. 风险：跨模块循环依赖。
   控制：只允许通过 `api.ts` 与 ports 协作。
2. 风险：消息渲染迁移导致 UI 回归。
   控制：先引入 `MessageContentHost`，再切换调用，最后删除旧分支。
3. 风险：目录重命名带来大规模冲突。
   控制：按阶段拆分 PR，避免跨阶段并行改同一文件。

## 6. 建议执行顺序（PR 维度）

1. PR-1：停用 ESLint + 文档重写
2. PR-2：目录骨架 + `api.ts` 入口统一
3. PR-3：`account` 合并
4. PR-4：`server-connection` 合并
5. PR-5：`chat` 重构（含 `message-flow/message` 抽象）
6. PR-6：`files` 语义拆分 + `shared/file-transfer`
7. PR-7：`shared/platform` 上移 + 全量清理
8. PR-8：验收收尾
