# 文档规范

本文档用于约束本仓库的长期维护文档写作与归档方式，目标是：

- 文档可持续维护（易发现、易更新、易评审）
- 信息不过期（标注适用范围与更新时间）
- 与代码演进一致（架构/协议/插件契约可追踪）

## 1. 归档位置与范围

- 长期维护文档：必须放在 `docs/` 下。
- 临时记录/一次性备忘：允许放在 PR/Issue/提交信息中，不要求进入 `docs/`。
- 根目录仅保留入口类文件（如 `README.md`）或“重定向说明”（指向 `docs/` 的真实文档）。

## 2. 命名规范（中文优先）

- `docs/` 下所有文档文件名使用中文，尽量表达主题（避免 `misc.md`、`temp.md`）。
- 可以使用数字前缀用于排序，例如：
  - `01-项目简介.md`
  - `02-架构设计.md`
- 需要记录时间的文档，建议使用日期前缀：
  - `2026-01-27-插件契约变更记录.md`
- 文件名避免特殊符号（建议仅使用：中文、数字、短横线 `-`）。

## 3. 文档结构模板

除索引类文档外，建议使用统一结构（可按需删减）：

1) **背景与目标**：为什么写、要解决什么问题  
2) **范围**：适用模块/版本/平台（例如仅前端、仅 Tauri）  
3) **设计/约定**：关键决策、约束、依赖关系  
4) **使用/流程**：如何使用、如何验证  
5) **变更记录**：关键变更点（可链接到 PR/commit）  
6) **相关链接**：指向代码位置或其他文档

## 4. 更新与评审

- 影响架构/协议/插件契约的改动：必须同时更新对应 `docs/` 文档。
- 文档应尽量与代码路径互相引用（使用相对链接）。
- 大改动建议在文档中增加“变更记录”小节，写明原因与迁移步骤。

## 5. 目录建议（可逐步落地）

可按主题逐步分目录（非强制）：

- `docs/开发/`：开发指南、环境、调试
- `docs/架构/`：架构设计、分层约束、依赖规则
- `docs/协议/`：TCP/HTTP 协议、消息格式
- `docs/插件/`：插件契约、能力声明、扩展点
- `docs/决策记录/`：重要决策（ADR）
