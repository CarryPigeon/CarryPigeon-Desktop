## 实施计划

### 公测版迭代阶段（v0.4.0 → public beta）

#### 阶段 1：稳定性与性能基线（已完成）
1. ✅ 建立 `isPerformanceMonitoringEnabled()` 统一开关，release 默认关闭
2. ✅ 启动时间埋点（`Action: app_startup_ready`）受开关控制
3. ✅ 内存监控（`memoryMonitor.ts`）受开关控制，release 不启动采样
4. ✅ 设置页新增"诊断模式"手动开关
5. ✅ `pnpm benchmark` 输出 JSON/Markdown 报告，支持回归检测
6. ✅ dev-only 内存长测入口（`memoryLongTest.ts`），release 产物不包含

#### 阶段 2：交互性能工程化（已完成）
1. ✅ 替换手写防抖为 `rateLimit.ts` 工具
2. ✅ 搜索输入统一使用 `rateLimit.debounce` / `debounceAsync`（ContactsPage、DomainCatalogPage、SearchPanel、ForwardChannelDialog）
3. ✅ chat 刷新函数（`refreshChannels`、`refreshChannelLatestPage`、`refreshMembersRail`）使用 `asyncDedupe` 去重
4. ✅ Web Worker 试点：消息排序自适应 Worker 化，合并后消息数 > 2000 时 offload 到 Worker，小数据量 fallback 主线程

**关键产出**：
- `src/shared/utils/rateLimit.ts` 统一防抖/节流/异步去重 API，覆盖搜索与刷新场景。
- `src/features/chat/adaptiveMessageSorter.ts` / `adaptiveMergeMessages.ts` 实现消息排序 Web Worker 自适应逻辑。
- 聊天、联系人、域目录、搜索面板、转发频道对话框等高频交互组件完成防抖与去重改造。

#### 阶段 3：后端 Rust 优化
1. 文件 IO 流式化（上传/下载/语音/截图）
2. 配置缓存（`get_config_value` TTL 缓存）
3. 高频配置写入批量 flush

#### 阶段 4：文档与发布准备
1. 同步 `release-readiness-checklist.md`
2. 补全 `CHANGELOG.md` v0.4.0 条目
3. 最终质量验证与公测发布

## 成功指标

- 应用启动时间 < 3秒（通过 `Action: app_startup_ready` 量化，诊断模式可观测）
- 消息列表滚动帧率 > 60fps
- 内存使用稳定在合理范围（dev-only 长测 + 内存监控）
- 编译时间减少30%

## 风险评估

### 高风险
- 大规模性能优化可能引入新问题
- 国际化扩展可能遗漏边缘情况

### 中风险
- 测试覆盖不足可能导致回归问题
- 文档不完整影响用户使用

### 低风险
- 小幅度的界面优化
- 文档补充和完善

## 总结

CarryPigeon Desktop项目具有良好的架构基础和代码质量，已经实现了核心聊天功能。通过系统化的性能优化、用户体验改进和测试完善，可以将其提升到可发布版本。

重点关注：
1. **性能优化**: 启动速度、渲染性能、资源管理
2. **用户体验**: 国际化、错误处理、界面交互
3. **质量保证**: 测试覆盖、稳定性验证
4. **发布准备**: 文档、变更日志、发布流程

通过分阶段实施，每个阶段都有明确的里程碑和验收标准，确保项目稳步推进到可发布状态。
